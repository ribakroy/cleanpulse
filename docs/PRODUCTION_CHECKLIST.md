# צ'קליסט מוכנות לייצור (Production Readiness Checklist)

מסמך זה משמש כצ'קליסט בדיקות ידניות ובקרת איכות לפני, במהלך ולאחר כל פריסה של מערכת CleanPulse ל-Production.

---

## 🟥 שלב 1: בדיקות לפני פריסה (Pre-Deployment)
* [ ] **הרצת בדיקות איכות מקומיות מלאות**:
  יש להריץ מקומית את הפקודה המאוחדת:
  ```bash
  npm run check
  ```
  ודא שאין שגיאות Lint, שאין שגיאות טיפוסים של TypeScript, והבנייה מסתיימת בהצלחה מלאה.
* [ ] **בדיקת פגיעויות אבטחה ברכיבים**:
  הרץ בדיקת אבטחה של התלויות:
  ```bash
  npm audit
  ```
  ודא שאין פגיעויות קריטיות (Critical/High). במקרה של פגיעויות Moderate/Low, ודא שהן מתועדות ושאינן משפיעות על קוד שרת שחשוף ללקוח.
* [ ] **אימות משתני סביבה סודיים**:
  ודא שקובץ ה-`.env.local` אינו נמצא תחת מעקב Git (מופיע ב-`.gitignore`).
  ודא שאין בקוד הפרויקט מפתחות גישה או טוקנים קשיחים (Hardcoded Secrets).
* [ ] **בדיקת גישה ל-GitHub Data Repo**:
  ודא שטוקן הגישה הקיים מורשה מול ריפו הנתונים הפרטי ושתקפותו אינה עומדת לפוג בקרוב.

---

## 🟧 שלב 2: הגדרה במהלך הפריסה (During Deployment)
* [ ] **הגדרת משתני הסביבה ב-Vercel**:
  ודא שכל המשתנים הבאים מוגדרים בממשק הניהול של Vercel (ראה פירוט במדריך ה-Vercel):
  * `NEXT_PUBLIC_APP_URL`
  * `AUTH_SECRET`
  * `DATA_ADAPTER` (מכוון ל-`github`)
  * `GITHUB_DATA_OWNER`, `GITHUB_DATA_REPO`, `GITHUB_DATA_BRANCH`, `GITHUB_DATA_TOKEN`
  * `EMAIL_PROVIDER` (מוגדר ל-`mock` עד אישור דומיין)
* [ ] **מעקב אחרי ה-Build Log**:
  בעת ביצוע ה-Deployment ב-Vercel, עקוב אחר ה-Log וודא ששלב ה-Build וההידור מסתיימים ללא שגיאות.

---

## 🟩 שלב 3: בדיקות מיד לאחר הפריסה (Post-Deployment QA)
לאחר סיום ה-Deployment, יש לעבור על הבדיקות הבאות בכתובת האתר האמיתית:

### 1. בדיקות גישה ואבטחה
* [ ] **בדיקת ה-Health endpoint**:
  גש ל-`/api/health`. ודא שחזר `status: "ok"` ו-`githubRepoReachable: true`.
  ודא שלא נחשפו שדות רגישים כגון `GITHUB_DATA_TOKEN` או `AUTH_SECRET` בתוך ה-JSON של התגובה.
* [ ] **בדיקת חסימת אזור ניהול ללא התחברות**:
  נסה לגשת ישירות לכתובת `/admin/dashboard`. ודא שהמערכת מבצעת הפניה (Redirect) אוטומטית לדף ה-`/login` ומונעת כניסה.

### 2. בדיקת מערכת ניהול (Admin Panel)
* [ ] **בדיקת התחברות (Login)**:
  התחבר עם המשתמש המנהל המוגדר בריפו הנתונים. ודא שההתחברות מצליחה והמערכת מפנה אותך לדשבורד.
* [ ] **בדיקת נתוני דשבורד**:
  ודא שכרטיסי ה-KPI (דיווחים, זמני טיפול, SLA) והגרפים מציגים את נתוני האמת מתוך ה-Data Repo בגיטהאב.
* [ ] **בדיקת עריכת הגדרות (CRUD)**:
  כנס לאזור ההגדרות (סניפים, שירותים, מסכים) ונסה לעדכן שם של סניף או להוסיף שירותים חדשים.
  ודא שהעדכון נשמר במערכת ומשתקף מיד.
* [ ] **אימות כתיבה לריפו הנתונים**:
  לאחר ביצוע השינוי, כנס ל-GitHub לתוך הריפו `cleanpulse-data` וודא שהקובץ הרלוונטי עודכן ושמתווסף Commit חדש של השינוי.

### 3. בדיקת ממשקי דיווח ציבוריים (Kiosk & QR)
* [ ] **בדיקת דף הטאבלט (Kiosk)**:
  כנס לדף המסכים בניהול, העתק את קישור ה-Kiosk ופתח אותו.
  ודא שהדף הציבורי עולה, חוסם את סרגל הניווט ומאפשר דיווח על תקלות.
  נסה לדווח על תקלה (למשל: "נייר חסר") וודא שהמערכת מציגה מסך תודה ומחזירה אותך למסך הדיווח לאחר 3 שניות.
* [ ] **בדיקת דף המובייל (QR)**:
  פתח את קישור ה-QR של המסך במובייל (או סרוק אותו).
  ודא שהממשק מאפשר דירוג כוכבים ושליחת דיווח מהיר.
* [ ] **בדיקת עדכון אות חיים (Last Seen)**:
  לאחר פתיחת ה-Kiosk, חזור למסך ניהול המסכים בלוח הבקרה וודא ששדה "נראה לאחרונה" עודכן לזמן הנוכחי (מראה על אות חיים תקין מהמסך).

### 4. אימות דיווחים ודוחות
* [ ] **בדיקת רישום תקלה חדשה במערכת**:
  לאחר שליחת דיווח מה-Kiosk/QR, ודא שהתקלה מופיעה מיד בדשבורד (תחת "דורש טיפול עכשיו") ובמסך הדיווחים.
* [ ] **בדיקת לוג התראות (Email Alert Log)**:
  בדוק את רשומת ה-Incident שנוצרה. ודא שתגית ההתראה מציגה `mock_sent` (מראה שהמערכת זיהתה את הנמענים ושלחה לוג התראה מדומה).
* [ ] **בדיקת ייצוא דוחות (CSV Export)**:
  כנס לדף הדוחות (`/admin/reports`), לחץ על כפתור "ייצוא ל-CSV" והורד את הקובץ.
  פתח את הקובץ באקסל או ב-Google Sheets. ודא שהטקסטים בעברית מוצגים בצורה קריאה ותקינה (ללא ג'יבריש) ושאין דליפת נתונים סודיים בקובץ.

---

## 🟨 שלב 4: מנגנון שחזור לאחור (Rollback / Disaster Recovery)
במקרה של תקלה חמורה לאחר פריסה או כשל נתונים:
1. **שחזור גרסת אפליקציה**:
   בממשק Vercel, גש ל-Deployments, בחר את הגרסה הקודמת התקינה ולחץ על **Rollback**. פעולה זו תחזיר מיד את גרסת הקוד הקודמת לפעילות תוך שניות בודדות.
2. **שחזור נתונים (Data Rollback)**:
   אם נתוני גיטהאב נפגמו או נמחקו בטעות, גש ל-Repository של הנתונים `cleanpulse-data` בגיטהאב והשתמש ב-Git History כדי לבצע revert ל-Commit האחרון התקין או לשחזר את הקבצים.
