# CleanPulse MVP PRD

## 1. תקציר מוצר

CleanPulse הוא מוצר דיווח בזמן אמת על מצב שירותים בעסקים.
המטרה ב־MVP היא לאפשר ללקוח הקצה לדווח בשתי נגיעות על בעיה או דירוג כללי, ולתת לעסק ממשק ניהול פשוט בעברית מלאה לניהול טיפול, התראות ומדידה תפעולית.

הנחת היסוד של ה־MVP:

- מסך טאבלט אנדרואיד מוצב בכניסה או ביציאה מהשירותים.
- כל מסך משויך למסך ציבורי אחד (`screen`) עם קישור פומבי מבוסס `publicToken`.
- קוד QR פותח קישור גיבוי ציבורי לאותו מסך.
- כל דיווח יוצר `incident`.
- המערכת שולחת התראת מייל לפי היררכיית נמענים מוגדרת מראש.
- מנהלים, אדמינים, בעלי עסק ומנקים עובדים דרך אתר רספונסיבי אחד.

## 2. הבעיה העסקית

עסקים יודעים על בעיות בשירותים מאוחר מדי או בכלל לא.
גם כשיש דיווח, הוא לא מחובר למיקום מדויק, לא מתועד, ולא מאפשר להבין זמן תגובה, עומסים חוזרים או ביצוע צוות.

ה־MVP פותר ארבע בעיות:

1. דיווח מיידי מהשטח עם חיכוך נמוך מאוד.
2. שיוך מלא של כל דיווח לארגון, סניף, אזור, שירותים ומסך.
3. התראה מהירה לגורם הרלוונטי.
4. תיעוד סטטוס וזמני טיפול לצורך ניתוח מגמות.

## 3. קהלי יעד

### משתמשי קצה

- מבקר/לקוח בשירותים.
- לא מזדהה.
- צריך ממשק גדול, ברור ומהיר.

### משתמשי ניהול

- `owner`: בעל העסק או הנהלה.
- `admin`: משתמש אדמיניסטרטיבי ברמת הארגון.
- `manager`: מנהל סניף או אזור.
- `cleaner`: איש צוות מבצע.

## 4. מטרות MVP

- לאפשר דיווח ציבורי מהיר דרך טאבלט או QR.
- לאפשר טיפול באירועים עם סטטוסים ברורים.
- לאפשר קבלת התראות מייל לפי מיקום.
- לאפשר צפייה בדשבורד, אירועים פתוחים, היסטוריה ודוחות בסיסיים.
- לשמור את הדאטה ב־GitHub Data Repo פרטי, בלי מסד נתונים קלאסי בשלב הראשון.
- להפריד לחלוטין בין Repo של האפליקציה ל־Repo של הדאטה.

## 5. מחוץ ל־MVP

- WhatsApp, SMS או Push.
- אפליקציה נייטיב.
- חיבור ל־Supabase או Postgres בגרסה הראשונה.
- SLA מתקדם עם הסלמות אוטומטיות.
- תמונות, וידאו או קבצים מצורפים.
- רב־לשוניות מעבר לעברית.
- BI מתקדם, חיזוי או מנוע חוקים מתקדם.

## 6. חוויית מוצר ועיצוב

- עברית מלאה ו־RTL מלא.
- מראה נקי, פרימיום, SaaS מודרני.
- שפת צבע מותגית: כחול מים נקי עם primary `#1E88E5`.
- במסך הציבורי: כפתורים גדולים מאוד, מרווחים גדולים, ניגודיות גבוהה, טקסט חד.
- בממשק הניהול: מינימום עומס, טבלאות פשוטות, סטטוסים ברורים, CTA אחד מרכזי בכל מסך.
- הממשק מיועד לעסק לא טכני, לכן שפה מוצרית פשוטה, בלי מונחים פנימיים.

## 7. סוגי דיווחים ב־MVP

המערכת תתמוך לפחות בסוגי הדיווח הבאים:

- חסר נייר
- חסר סבון
- לא נקי
- ריח לא טוב
- פח מלא
- תקלה באסלה
- תקלה בכיור
- חוות דעת כללית: 1–5 כוכבים

הערה מוצרית:
דירוג כוכבים יישמר כ־`rating` על גבי `incident`, נפרד מסוגי הבעיה.

## 8. User Flows מלאים

### Flow A: דיווח ציבורי מטאבלט

1. המבקר רואה מסך ציבורי במסלול `/k/[publicToken]`.
2. הוא לוחץ על בעיה או דירוג.
3. המערכת מאמתת שה־`publicToken` שייך למסך פעיל.
4. נוצר `incident` עם שיוך מלא ל־`organizationId`, `branchId`, `restroomId`, `screenId`.
5. נוצר `activity_log` מסוג `incident_created`.
6. המערכת פותרת נמענים לפי היררכיית התראות.
7. נשלח מייל אם נמצאו נמענים.
8. מוצג מסך תודה קצר וחוזרים למסך הראשי אוטומטית.

### Flow B: דיווח ציבורי דרך QR

1. המבקר סורק QR.
2. נפתח מסלול `/b/[publicToken]` עם אותו מסך דיווח, מותאם למובייל.
3. ממשיכים בדיוק כמו ב־Flow A.

### Flow C: קבלת התראה וטיפול ראשוני

1. נמען מקבל מייל עם פרטי האירוע.
2. נכנס למערכת הניהול.
3. רואה אירוע פתוח בדשבורד.
4. משנה סטטוס ל־`acknowledged` או `in_progress`.
5. המערכת שומרת מי שינה, מתי, ומה השתנה.

### Flow D: סגירת אירוע

1. עובד רלוונטי מטפל בבעיה בפועל.
2. נכנס לאירוע.
3. מסמן `resolved` או `dismissed`.
4. המערכת מחשבת `timeToClose`.
5. האירוע עובר לדוחות ולהיסטוריה.

### Flow E: דוחות

1. מנהל בוחר טווח: יומי, שבועי או חודשי.
2. המערכת מציגה:
   - כמות דיווחים
   - התפלגות לפי סוג תקלה
   - התפלגות לפי סניף / שירותים / מסך
   - זמן תגובה וזמן סגירה
   - כמות אירועים פתוחים מול סגורים
   - דירוג ממוצע

### Flow F: Setup ראשוני

1. `owner` או `admin` יוצר ארגון.
2. מוסיף סניפים.
3. מוסיף אזורי שירותים.
4. מוסיף מסכים.
5. לכל מסך נוצר `publicToken` ארוך ואקראי.
6. המערכת מייצרת קישור טאבלט וקישור QR.
7. האדמין מגדיר נמעני התראות ברמות השונות.
8. הארגון מוכן לעבודה.

## 9. Scope פונקציונלי ל־MVP

### Public Reporting

- מסך דיווח ציבורי מלא.
- תמיכה בכפתורי תקלות קבועים.
- תמיכה בדירוג כוכבים.
- מסך תודה והחזרה אוטומטית למסך הבית.

### Admin

- התחברות עם אימייל + סיסמה.
- דשבורד בסיסי.
- רשימת אירועים.
- עמוד אירוע בודד.
- שינוי סטטוס ושיוך מטפל.
- תיעוד מי קיבל, מי טיפל ומי סגר.
- ניהול סניפים, שירותים, מסכים, משתמשים ונמעני התראות.
- דוחות יומי / שבועי / חודשי.

### Notifications

- התראות מייל בזמן יצירת אירוע.
- `notification_logs`.
- תיעוד כשלי שליחה או מצב `no_recipients`.

## 10. Edge Cases

- `publicToken` לא תקף או מנוטרל.
- דיווח כפול עקב לחיצות חוזרות.
- אין נמענים בכלל למסך/שירותים/סניף/ארגון.
- כשל ב־GitHub API בזמן כתיבה.
- כשל בשליחת מייל אחרי שהאירוע כבר נוצר.
- משתמש עם הרשאה נמוכה מנסה לפתוח נתונים מחוץ לסקופ שלו.
- מחיקת סניף/מסך שיש להם היסטוריית אירועים.
- שינוי שם סוג תקלה אחרי שכבר נוצרו אירועים היסטוריים.

## 11. מדדי הצלחה ל־MVP

- זמן דיווח ממוצע של פחות מ־5 שניות.
- פחות מ־1% כשלים ביצירת אירוע.
- 100% שיוך של כל אירוע לארגון, סניף, שירותים ומסך.
- שליחת התראה תקינה בכל אירוע שיש לו נמענים.
- אפשרות לצפות בדוח יומי/שבועי/חודשי לכל ארגון פעיל.

## 12. Definition of Done ברמת מוצר

- ניתן להקים ארגון עם לפחות סניף אחד, שירותים אחד ומסך אחד.
- ניתן לבצע דיווח ציבורי אמיתי דרך קישור פומבי.
- כל דיווח נשמר ב־Data Repo ולא ב־App Repo.
- data changes אינם מפעילים deployment.
- התראות מייל עובדות במסלול production.
- ממשק הניהול מאפשר טיפול וסגירה של אירועים.
- הדוחות מציגים נתונים נכונים על בסיס אירועים שנשמרו.
- כל הממשק ב־RTL ובעברית מלאה.

## 13. Post-MVP Scope

- SLA והסלמות.
- תזכורות על אירועים פתוחים.
- תמיכה ב־SMS / WhatsApp / Push.
- יכולות אנליטיקה עמוקות יותר.
- multi-tenant scale עם DB רלציוני.
- תמיכה במצב offline חלקי בטאבלט.
