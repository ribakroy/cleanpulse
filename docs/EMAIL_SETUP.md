# הגדרת מערכת התראות מייל (Email Setup)

מערכת CleanPulse כוללת מנגנון מובנה לשליחת התראות מייל בזמן אמת לנמענים מוגדרים בעת קבלת דיווחים ציבוריים (תקלות או דירוגים). 

התשתית תומכת בשני ספקי שליחה (Providers):
1. **Mock Provider** (סביבת פיתוח/בדיקות).
2. **Resend Provider** (סביבת ייצור).

---

## משתני סביבה (Environment Variables)

להפעלת המערכת יש להגדיר את משתני הסביבה הבאים בקובץ `.env` (או `.env.local`):

```bash
# ספק המייל הפעיל: mock או resend (ברירת מחדל: mock)
EMAIL_PROVIDER=mock

# מפתח ה-API של Resend (נדרש רק אם EMAIL_PROVIDER=resend)
RESEND_API_KEY=re_123456789

# כתובת השולח המורשית ב-Resend (למשל: alerts@yourdomain.com)
EMAIL_FROM=CleanPulse Alerts <onboarding@resend.dev>

# כתובת למענה (Reply-To) - אופציונלי
EMAIL_REPLY_TO=support@yourdomain.com

# כתובת האפליקציה (לצורך יצירת קישורים במייל)
NEXT_PUBLIC_APP_URL=http://localhost:3000
```

---

## ספקי שליחה (Providers)

### 1. ספק מדומה (Mock Provider)
* **מתי לפעיל**: סביבת פיתוח מקומית ובדיקות אינטגרציה.
* **התנהגות**:
  - אינו מבצע קריאות רשת ואינו שולח מיילים בפועל.
  - מדפיס ל-Console פלט מפורט של המייל (נמענים, נושא, גוף ההודעה בטקסט פשוט).
  - מייצר רשומת log עם מזהה הודעה מדומה וסטטוס `mock_sent`.

### 2. ספק Resend (Resend Provider)
* **מתי להפעיל**: סביבת ייצור (Production / Staging).
* **התנהגות**:
  - מבצע קריאת API רשמית לשרתי Resend לשליחת המייל לנמען.
  - במידה והשליחה מצליחה, מייצר רשומת log בסטטוס `sent`.
  - במידה והשליחה נכשלת (בעיית רשת, מפתח API לא תקין וכו'), המערכת תתפוס את השגיאה ותייצר רשומת log בסטטוס `failed` המכילה את פירוט השגיאה ב-`errorMessage`.
  - **הגנה מקריסות**: שגיאה בשליחת המייל לא תפיל את יצירת ה-Incident. הדיווח הציבורי יישמר במלואו, והשגיאה תתועד רק בלוגים לצורך ניטור.

## לוגיקת פתרון נמענים היררכית (Recipient Resolution)

כאשר מתקבל דיווח חדש עבור מסך מסוים, המערכת מחפשת נמענים פעילים בהיררכיה הבאה:

1. **נמענים ברמת מסך** (`scopeType = screen`): נמענים המשויכים ישירות לטאבלט הפיזי שבו בוצע הדיווח.
2. **נמענים ברמת אזור שירותים** (`scopeType = restroom`): במידה ולא נמצאו נמענים למסך, נשלח לכל מי שמנוי על אזור השירותים.
3. **נמענים ברמת סניף** (`scopeType = branch`): במידה ולא נמצאו למעלה, נשלח למנהלי הסניף.
4. **נמענים ברמת ארגון** (`scopeType = organization`): במידה ולא נמצאו באף רמה, נשלח לנמענים הראשיים של הארגון.
5. **אין נמענים**: במידה ולא מוגדר אף נמען באף אחת מהרמות, המערכת תיצור רשומת log יחידה עם סטטוס `no_recipients`.

*הערה: המערכת שולחת מייל נפרד לכל נמען ומייצרת רשומת log ייעודית עבורו ב-`notification_logs`.*

## תבנית המייל (Email Templates)

* **שפה**: עברית מלאה בכיווניות מימין לשמאל (RTL).
* **עיצוב**: HTML נקי ומודרני מבוסס inline styles, בצבעי כחול מים של מותג CleanPulse.
* **תוכן**:
  - **נושא (Subject)**:
    - תקלה: `"דיווח שירותים חדש — {שם התקלה} | {שם הסניף}"`
    - דירוג: `"דירוג שירותים חדש — {כוכבים}/5 | {שם הסניף}"`
  - **גוף המייל**: טבלה מפורטת הכוללת את סוג הדיווח, הסניף, אזור השירותים, שם המסך, הבעיה/דירוג המדויק, זמן הדיווח, ומקור הדיווח (טאבלט/QR).
  - **לחצן קריאה לפעולה (CTA)**: כפתור מעוצב המוביל ללוח הניהול.
  - **טקסט חלופי (Text Fallback)**: מיוצר במקביל עבור לקוחות מייל שאינם תומכים ב-HTML.

## הוראות לבדיקה ידנית

1. ודא שהגדרת לפחות נמען אחד פעיל בממשק הניהול (`/admin/recipients`) או שהרצת את ה-seed המקומי:
   ```bash
   npm run seed:local
   ```
2. נווט למסך טאבלט ציבורי ובצע דיווח (למשל: `/k/cp_public_demo_floor_1_Q7h9L2m5R8x1T6u3W4y9Z2a7B1c6D8e0`).
3. בדוק את פלט ה-Terminal/Console של השרת - ודא שמופיעה הדפסה ברורה של גוף המייל של ה-Mock Provider.
4. היכנס לממשק הניהול של הדיווחים (`/admin/incidents`) - ודא שליד הדיווח החדש מופיעה תווית כחולה: `נשלח (Mock)`.
5. שנה את הגדרות ה-recipients כך שלא יהיו נמענים פעילים בארגון, בצע דיווח חדש, וודא שבלוח הניהול מופיעה תווית אפורה: `אין נמענים להתרעה`.
6. כדי לבדוק טיפול בשגיאות: שנה ב-`.env` את משתנה הסביבה ל-`EMAIL_PROVIDER=resend` (ללא מפתח API), בצע דיווח, וודא שהדיווח נקלט בהצלחה אך בלוח הניהול מופיעה תווית אדומה: `התראת מייל נכשלה`.
