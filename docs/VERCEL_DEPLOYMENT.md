# מדריך פריסה והגדרת Vercel (Vercel Deployment Guide)

מדריך זה מפרט את הצעדים וההנחיות לפריסה בטוחה ותקינה של אפליקציית CleanPulse בפלטפורמת Vercel.

---

## 1. חיבור ה-Repository ב-Vercel
* **ריפו האפליקציה**: יש לחבר ל-Vercel את ה-Repository של האפליקציה בלבד: `ribakroy/cleanpulse`.
* **ריפו הנתונים**: **אל תחבר** את ה-Repository הפרטי של הנתונים (`ribakroy/cleanpulse-data`) לפרויקט ה-Vercel שלך. הנתונים נשארים מופרדים לחלוטין ונגישים רק שרתית דרך ה-GitHub API.

---

## 2. הגדרות הפרויקט ב-Vercel (Project Settings)
בעת הקמת הפרויקט ב-Vercel, ודא שהגדרות הבנייה הן הבאות:
* **Framework Preset**: `Next.js`
* **Root Directory**: `./` (שורש הפרויקט)
* **Build Command**: `npm run build`
* **Install Command**: `npm install`

---

## 3. הגדרת משתני סביבה (Environment Variables)
לפני הלחיצה על "Deploy", יש להגדיר את משתני הסביבה הבאים ב-Vercel. 
(עבור משתנים סודיים, מומלץ להסתיר את ערכם בממשק Vercel).

| שם המשתנה | ערך מומלץ / הנחיות | הערות |
| :--- | :--- | :--- |
| `NEXT_PUBLIC_APP_URL` | `https://your-app-domain.vercel.app` (ראה הערה למטה) | כתובת האתר האמיתית |
| `AUTH_SECRET` | מחרוזת אקראית חזקה (למשל של 32 תווים) | משמש לחתימת עוגיות ה-JWT |
| `DATA_ADAPTER` | `github` | שימוש ב-GitHub כמסד הנתונים |
| `GITHUB_DATA_OWNER` | `ribakroy` | שם המשתמש ב-GitHub |
| `GITHUB_DATA_REPO` | `cleanpulse-data` | שם ה-Repository של הנתונים |
| `GITHUB_DATA_BRANCH` | `main` | ענף הנתונים הראשי |
| `GITHUB_DATA_TOKEN` | `github_pat_...` (הטוקן המעודכן) | טוקן Fine-grained עם הרשאות כתיבה/קריאה לנתונים |
| `EMAIL_PROVIDER` | `mock` (בשלב הראשון) | שימוש בשרת התראות מדומה ב-Console |
| `RESEND_API_KEY` | *(להשאיר ריק בשלב ראשון)* | מפתח ה-API של Resend להתראות אמיתיות |
| `EMAIL_FROM` | *(להשאיר ריק בשלב ראשון)* | כתובת הדוא"ל השולחת המאומתת ב-Resend |
| `EMAIL_REPLY_TO` | *(להשאיר ריק בשלב ראשון)* | כתובת למענה עבור מנהלים |
| `DEMO_MODE` | `false` | מונע הצגת פרטי התחברות של דמו בדף הכניסה |

### ⚠️ שימו לב לגבי `NEXT_PUBLIC_APP_URL`
בפריסה הראשונית, ייתכן שלא תדע עדיין את ה-URL הסופי ש-Vercel תקצה לפרויקט (או ה-Custom Domain שלך).
1. בשלב ראשון, ניתן להגדיר את המשתנה ככתובת זמנית כלשהי או להשתמש בכתובת ברירת המחדל.
2. לאחר שהפרויקט נפרס בהצלחה ו-Vercel הקצתה עבורו דומיין (למשל `cleanpulse.vercel.app`), יש להיכנס ל-**Project Settings -> Environment Variables** ב-Vercel, לעדכן את ערך `NEXT_PUBLIC_APP_URL` לכתובת האמיתית (כולל `https://`), ולבצע פריסה מחדש (Redeploy) כדי שהשינוי ייכנס לתוקף.
*זהו שלב קריטי כדי שה-QR Codes במסכים ובדוחות יקשרו לממשק הדיווח הציבורי האמיתי.*

---

## 4. בדיקות ואימות לאחר הפריסה (Post-Deployment Verification)
לאחר סיום הפריסה ב-Vercel, בצע את הבדיקות הבאות לפי הסדר:
1. **בדיקת בריאות (/api/health)**:
   גש לכתובת `https://your-domain.com/api/health`. ודא שאתה מקבל קובץ JSON עם סטטוס `200 OK` שמציג `status: "ok"` ו-`githubRepoReachable: true`.
2. **דף ההתחברות (/login)**:
   ודא שדף ה-Login עולה ללא שגיאות 500, ושהסבר הדמו והסיסמאות אינם מופיעים (כאשר `DEMO_MODE=false`).
3. **התחברות ולוח ניהול (/admin/dashboard)**:
   נסה להתחבר עם המשתמש הניהולי המוגדר בנתונים. ודא שהחיבור מצליח, שהעוגייה נוצרת בצורה מאובטחת, ושהדשבורד מציג את נתוני הדיווחים מתוך גיטהאב.
4. **בדיקת קישורי דיווח (Kiosk & QR)**:
   גש למסכי ניהול ה-Screens, העתק קישור Kiosk או QR פתוח, וודא שהמערכת מאפשרת לפתוח אותם בצורה ציבורית ולדווח על תקלות או דירוג כוכבים.

---

## 5. כללי זהירות חשובים (Do's & Don'ts)
* ❌ **אל תעביר את ה-Repository של הנתונים ל-Vercel**: אל תקשר את הריפו `cleanpulse-data` או תבנה אותו. הוא חייב להישאר מופרד ונסתר.
* ❌ **אל תחשוף את ה-GITHUB_DATA_TOKEN**: הטוקן לעולם אינו מוגדר עם תחילית `NEXT_PUBLIC_` והוא נגיש אך ורק לקוד שרת (API Routes ו-Server Actions).
* ❌ **אל תפעיל שליחת מיילים דרך Resend ב-Production לפני אימות דומיין**: שליחה דרך דומיינים לא מאומתים ב-Resend תיחסם או תגיע ישירות לספאם. ודא קודם כל שסיימת את תהליך ה-Domain Verification ב-Resend.

---

## 6. מגבלות ארכיטקטוניות ונקודות לתשומת לב
* **מגבלות קצב של GitHub API**:
  טוקן גישת GitHub מאומת מוגבל ל-5,000 בקשות בשעה. מכיוון שהאפליקציה מבצעת קריאות רקורסיביות ל-JSON files, הדבר מספיק בקלות לארגונים קטנים ובינוניים אך עשוי להוות צוואר בקבוק בארגונים גדולים עם מאות דיווחים בדקה.
* **Serverless Cold Starts**:
  קריאה ראשונה של פונקציות Serverless (לאחר זמן חוסר פעילות) עשויה לקחת בין 1.5 ל-3 שניות עקב תהליך האתחול מול ה-GitHub API.
* **ניהול התנגשויות בעריכה מקבילה (Concurrency)**:
  במקרה של כתיבה מקבילה של שני משתמשים לאותו קובץ, שכבת ה-GitHubDataAdapter משתמשת ב-SHA של הקובץ וכוללת מנגנון של עד 3 ניסיונות כתיבה חוזרים (Retry) לפתרון התנגשויות.
