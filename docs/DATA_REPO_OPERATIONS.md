# מדריך תפעול מאגר הנתונים (Data Repo Operations Guide)

מסמך זה מפרט את הנהלים התפעוליים והתחזוקה עבור מאגר הנתונים המבוזר של CleanPulse המבוסס על `cleanpulse-data` (GitHub כבסיס נתונים).

---

## 1. מבנה התיקיות והקבצים ב-Data Repo

מאגר הנתונים נשמר בפורמט JSON שטוח או מחולק לפי תאריכים בתוך תיקיית `data/`:

```text
data/
  organizations/                 # פרטי הארגון (רישיון, תוכנית, סטטוס)
    [org_id].json
  users/                         # משתמשי המערכת (מנהלים, צוות ניקיון)
    [user_id].json
  branches/                      # סניפי הארגון
    [branch_id].json
  restrooms/                     # אזורי שירותים משויכים לסניפים
    [restroom_id].json
  screens/                       # מסכי טאבלט ושאילתות
    [screen_id].json
  issue_types/                   # הגדרות תקלות ורמות חומרה (גלובלי)
    [issue_type_id].json
  notification_recipients/       # נמעני התראות (מנהלים/צוות) לפי סניפים
    [recipient_id].json
  incidents/YYYY/MM/             # דיווחים (מקונן לפי שנה וחודש UTC)
    [incident_id].json
  activity_logs/YYYY/MM/         # יומן פעולות שבוצעו (שינויי סטטוס)
    [log_id].json
  notification_logs/YYYY/MM/     # יומן שליחת התראות (מייל)
    [log_id].json
```

## 2. חוק בל יעבור: עריכה ידנית של קובצי JSON

> [!CAUTION]
> **אין לערוך קובצי JSON במאגר ישירות ב-GitHub ללא הבנה מעמיקה של מבנה הנתונים!**
> עריכה ידנית שגויה עלולה לשבור את התאימות לטיפוסי הנתונים ב-TypeScript (קומפילציה שרתית) ולהביא לקריסת האפליקציה.

### הנחיות לעריכה ידנית (במידה ואין ברירה):
* ודא שאתה משתמש ב-JSON ולידי (ללא פסיקים מיותרים בסוף מערכים/אובייקטים).
* אל תשנה שדות תאריכים (`createdAt`, `openedAt`) לפורמט שאינו ISO 8601 (לדוגמה: `2026-06-09T17:40:00.000Z`).
* אל תשנה את ה-`id` של הקובץ או את שם הקובץ עצמו (שם הקובץ חייב תמיד להיות זהה לשדה `id` בתוכו).
* ודא שהמזהים המקשרים (למשל `branchId` בתוך `restrooms`) קיימים בפועל בתיקייה המתאימה.

## 3. גיבוי ושחזור (Backup & Restore)

מכיוון שהנתונים מאוחסנים ב-Git Repository רגיל, הגיבוי הוא פשוט ומובנה:

### ביצוע גיבוי
1. שכפול פשוט (Clone) של המאגר למחשב מקומי או לשרת גיבויים:
   ```bash
   git clone git@github.com:ribakroy/cleanpulse-data.git
   ```
2. כל גרסה (Commit) ב-Git מהווה נקודת שחזור היסטורית מלאה.

### שחזור לנקודת זמן (Rollback)
אם אחד העדכונים גרם לבעיה בדאטה, ניתן לבצע Rollback דרך Git בקלות:
```bash
# הצגת היסטוריית השינויים האחרונים
git log --oneline

# חזרה לקומיט תקין קודם
git checkout [commit-sha] -- data/

# ביצוע Commit ו-Push של הדאטה המשוחזר
git commit -m "Rollback data to healthy state"
git push origin main
```

## 4. מניעת התנגשויות (Conflicts) ופתרונן

בשל עבודה עם GitHub API כבסיס נתונים ללא מנגנון Transactions מסורתי, עלולות להיווצר התנגשויות כתיבה כאשר שני משתמשים מעדכנים רשומה בו-זמנית:

1. **זיהוי התנגשות באפליקציה**: השרת יזרוק שגיאה עם קוד `GITHUB_WRITE_CONFLICT` (שגיאת HTTP 409 מגיטהאב).
2. **מנגנון Retry מובנה**: ה-`GitHubDataAdapter` כולל מנגנון Retry אוטומטי של עד 3 ניסיונות עבור עדכונים. הוא קורא מחדש את ה-SHA העדכני של הקובץ ומנסה לשלוח שוב.
3. **פתרון התנגשות ידנית (במקרים קיצוניים)**:
   - אם סקריפט ה-seed או האפליקציה נכשלים עקב קובץ נעול או SHA לא תואם, בצע `git pull` מקומי בתיקיית הנתונים, פתור את ה-Conflict בקובץ ה-JSON בדומה לקוד רגיל, ובצע Commit & Push.

## 5. אזהרות קריטיות (Warnings)

> [!IMPORTANT]
> - **לא להפוך את המאגר לציבורי (Never Make Public)**: מאגר הנתונים `cleanpulse-data` מכיל מידע רגיש הכולל שמות עובדים, כתובות מייל של לקוחות וצוות, והערות תפעוליות. הוא **חייב להישאר פרטי לחלוטין**.
> - **אין לאחסן סודות וטוקנים**: לעולם אל תכניס מפתחות API, סיסמאות פשוטות או טוקני גישה לתוך קובצי ה-JSON במאגר.
> - **לא לחבר את ה-Data Repo ל-Vercel**: מאגר האפליקציה הראשי (`cleanpulse`) הוא זה שמחובר ל-Vercel. חיבור `cleanpulse-data` ל-Vercel יגרום ל-Build ו-Deploy מיותרים בכל פעם שנפתח דיווח או משתנה סטטוס (מכיוון שכל פעולה יוצרת Commit חדש במאגר הנתונים).
> - **אין לערוך את ה-`passwordHash` ידנית**: שדה הסיסמה של המשתמשים מוצפן בעזרת אלגוריתם `bcrypt`. שינוי ידני של התווים בקובץ ה-JSON ינעל את המשתמש מחוץ למערכת.

## 6. נתיב הגירה עתידי ל-Postgres (Migration Path)

השימוש ב-GitHub כבסיס נתונים מתאים לשלב ה-MVP והפיילוט הראשוני בלבד. עם הגדלת נפח הפעילות (מעל 10 סניפים או מאות דיווחים ביום), מומלץ לעבור לבסיס נתונים יחסי כמו Postgres או Supabase.

### שלבי ההגירה:
1. **כתיבת Adapter חדש**: מימוש ממשק `DataAdapter` (הקיים ב-`lib/data/adapter.ts`) עבור Postgres/Prisma.
2. **פיתוח סקריפט הגירה**: סקריפט פשוט שיקרא את קובצי ה-JSON מתוך `cleanpulse-data` ויזין אותם (Insert) לטבלאות ב-Postgres.
3. **שינוי משתנה סביבה**: הגדרת `DATA_ADAPTER=postgres` (או הטיפוס החדש) בשרת ה-Production, ללא צורך בשינוי של שורת קוד אחת בשכבת התצוגה והלוגיקה של האפליקציה.
